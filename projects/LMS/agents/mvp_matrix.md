# MVP Team Matrix

This matrix groups common app-development roles by whether they are typically needed as separate responsibilities in an MVP.

| Role | MVP Need | Notes |
| --- | --- | --- |
| Product Manager | Must-have | Defines scope, priorities, and success criteria. |
| Project Manager / Scrum Master | Usually combined | Fold into product or technical lead unless coordination is complex. |
| Business Analyst | Usually combined | Fold into product/planning for MVP discovery. |
| UX Designer | Must-have | Needed to define the core flow and avoid a confusing product. |
| UI Designer | Usually combined | Often merged with UX for MVP unless visual polish is critical. |
| Frontend Developer | Must-have | Builds the user-facing app. |
| Backend Developer | Must-have | Builds APIs, business logic, and auth. |
| Software Architect | Usually combined | Architecture decisions are usually shared by backend and lead engineering. |
| QA Engineer | Must-have | Can be lightweight, but core flow testing is still required. |
| DevOps Engineer | Usually combined | MVPs usually need only the minimum deployment support. |
| Mobile Developer | Sometimes separate | Only separate if the MVP is mobile-first or platform-specific. |
| Database Specialist | Usually combined | Often handled by backend for MVP unless data complexity is high. |
| Security Specialist | Must-have | Can be lightweight, but basic security review is non-negotiable. |

## Interpretation

- `Must-have` means the responsibility needs coverage in the MVP.
- `Usually combined` means the work still matters, but it can often be handled by the same person or agent as another role.
- `Sometimes separate` means split it out only if the product or platform requires it.
